# 명세서 작성 가이드 (LLM 임의 채움 차단)

> **PoC 핵심**: 사용자 자연어 명세서를 받아 **그대로** 구현. LLM의 임의 판단을 어떻게 제어하는지가 교육 포인트.
>
> 이 문서는 **사용자(작성자)** 가 명세서를 작성하기 *전*에 빠뜨리기 쉬운 항목을 카테고리별로 점검하는 체크리스트입니다. 명세서가 모호하면 LLM은 *임의로 채워 넣습니다* — 그 결과 같은 prompt를 두 번 실행해도 다른 결과가 나옵니다.

## 핵심 원칙

1. **수치는 정확한 단위로** — "회복", "감소" 같은 동사만 적지 말고 *얼마 / 어떤 단위 / 어떤 주기*까지.
2. **시점은 frame 단위로** — "즉시" "곧" "잠시 후"는 모호. "그 frame에", "다음 frame에", "Nms 후에" 같이.
3. **금지 사항을 명시** — 명세에 *포함된 동작*뿐 아니라 *금지된 동작*도 명시. LLM이 "그럴듯해 보이는" 변형을 시도하지 못하게.
4. **예외 케이스 모두 열거** — boundary (0, max, null, 빈 배열, 동시 발생) 각각의 동작을 적기.
5. **모호 표현 금지** — 다음 단어를 명세서에서 발견하면 *반드시* 수치·시점·조건으로 치환:

| ❌ 모호 표현 | ✅ 구체화 |
|---|---|
| "적당히" / "잘" | 수치·임계값 명시 |
| "필요 시" / "가끔" | 조건문으로: "X 조건일 때만" |
| "보통" / "일반적으로" | 기본값 + 예외 분리 |
| "회복" / "감소" | "매 Nms마다 +K", "Nms 안에 K만큼" |
| "충분히" / "여유롭게" | 수치 + 단위 |
| "자연스럽게" / "부드럽게" | 알고리즘·easing 함수 명시 |
| "비슷하게" | 동일/다른 점을 구체적으로 |

## 1. 상태 변수 (state) — 초기값·변화·범위·시점

각 상태 변수마다 다음 4가지 모두 명시:

- [ ] **초기값** (게임 시작 / 페이지 로드 / 사용자 입력 직전)
- [ ] **범위** (min/max, clamp 여부)
- [ ] **변화량 + 단위 + 주기** (예: "+5 / 1초마다" 또는 "공격 시 -3")
- [ ] **변화 시점** (즉시 / 다음 frame / Nms 후)

**❌ 나쁜 예**:
> 플레이어는 MP가 있고 공격하면 줄어든다. 회복도 된다.

**✅ 좋은 예**:
> - MP: 초기값 100, 범위 [0, 100], `Math.min(mp + N, 100)`로 clamp.
> - **공격 시 감소**: 무기별 cost (성검 0 / 불꽃 5 / 번개 3 / 성광 8) 즉시 차감. cost > 현재 MP면 발사 *불가* (성검은 cost 0이라 항상 가능).
> - **회복**: 매 1초(1000ms)마다 +5. 단 *공격 직후 300ms cooldown 동안은 회복 X*. `setInterval` 또는 `useEffect` 안 setTimeout.

## 2. 이벤트 시점 — frame 단위 명시

"즉시" / "곧" / "잠시 후" 같은 표현 금지. 다음 중 하나로:

- 같은 frame 안에 (synchronous)
- 다음 render frame (1~16ms 후)
- 정확히 Nms 후 (setTimeout)
- 조건 충족 frame에 (조건 polling)

**❌ 나쁜 예**:
> HP가 0이 되면 게임이 끝난다.

**✅ 좋은 예**:
> HP가 0 또는 음수가 되는 *그 frame에* 즉시 `navigate('/result')` 호출. 추가 frame 진행 금지 (다음 update tick 진입 전에 cleanup). 진행 중인 적 spawn / 탄환 update 모두 중단.

---

## 3. 적 / NPC AI — 패턴·궤적·금지 사항

명세서에는 *허용 패턴*뿐 아니라 *금지 패턴*도 명시.

- [ ] 발사 주기 (정확한 ms)
- [ ] 발사 시점의 *조준 방식* (정지 직선 / 발사 순간 플레이어 위치 / 추적)
- [ ] 탄환 속도·방향·생존 시간
- [ ] **금지 동작** (유도탄 / 추적 / 변형 곡선 등)
- [ ] 사망 조건 (1발 즉사 / HP 누적 / 무적 시간)

**❌ 나쁜 예**:
> 적은 자기 무기로 주인공을 공격한다.

**✅ 좋은 예**:
> - **조준**: 발사 *시점*의 주인공 위치로 *직선* 발사. 발사 이후 궤적은 변하지 않음 (탄환 등록 시점에 dx/dy 계산해 frame마다 위치 update).
> - **🚫 금지**: 유도탄(homing) / 추적(tracking) / 곡선 궤적 / 발사 후 dx 재계산. 이 4가지는 *명시적 위반*.
> - **속도**: 250 px/s (또는 무기별 표 참조).
> - **주기**: 무기 표 ms 그대로. *임의 배수 금지*.
> - **사망**: 플레이어 탄환에 한 발 맞으면 즉시 제거. HP·무적시간 *없음*.

## 4. UI / HUD — 위치·크기·색·수치

화면 요소는 다음 5가지 모두 명시 (단, 게임 화면이 아닌 일반 form은 위치·크기 약식 가능):

- [ ] **위치** (top-left / 중앙 / `position: absolute; top: 10px; left: 20px`)
- [ ] **크기** (px / % / rem)
- [ ] **색** (HEX 정확값)
- [ ] **표시할 데이터·형식** (정수 / 소수점 N자리 / 단위 표시)
- [ ] **숨김·표시 조건** (항상 / 특정 state 시만)

**❌ 나쁜 예**:
> HP는 화면에 표시한다.

**✅ 좋은 예**:
> HP 게이지:
> - 위치: 화면 좌상단, `top: 16px; left: 16px`.
> - 크기: 폭 200px, 높이 16px.
> - 색: 채워진 영역 `#FF4D4D`, 배경 `#333333`, 외곽선 `#FFFFFF` 1px.
> - 형식: 게이지(`width: ${hp}%`) + 우측에 `"<현재>/<최대>"` 텍스트 (12px Sans, white).
> - 항상 표시. HP 변화 시 200ms transition.

## 5. CRUD / API — 입력·출력·예외

각 endpoint별로:

- [ ] 각 필드의 형식·예외 (validator 정의)
- [ ] 성공 응답 schema + status
- [ ] 모든 에러 케이스 + status + error message *정확 문자열*
- [ ] 처리 순서 (insert 전후, validation 단계)
- [ ] 동시성 (lock / transaction 필요 여부)
- [ ] 시드 데이터 (test scenario에서 사용할 값)

→ 본 PoC에서는 `rules/domain.md` §2 카탈로그가 표준. spec에 `pattern`/`example` 작성 시 카탈로그 *글자 그대로*.

## 6. 동시 발생·경합 조건

명세에 *동시에 일어날 수 있는* 두 이벤트가 있다면 처리 순서 명시:

**❌ 나쁜 예**:
> 적이 죽으면 score가 올라가고, score가 임계값 넘으면 다음 stage로 전환된다.

**✅ 좋은 예**:
> 적 제거 시 순서:
> 1. enemy.alive = false
> 2. score += stageScore[currentStage]
> 3. *score 갱신 직후 같은 frame에* threshold 검사: score >= STAGES[currentStage].scoreThreshold면 currentStage++
> 4. *stage 전환 시*: spawn pool 즉시 교체 (다음 spawn부터 적용, 이미 화면에 있는 적은 그대로)
> 
> 같은 frame에 2발이 동시에 적을 맞춘 경우: 먼저 처리된 탄환만 score 가산, 두 번째는 enemy.alive=false 보고 무시 (idempotent).

## 7. boundary / edge case 5종 (필수 체크)

- [ ] 0 / 음수 / 빈 입력 시 동작
- [ ] max / overflow 시 동작
- [ ] null / undefined 시 동작
- [ ] 동시 발생 시 처리 순서
- [ ] 네트워크 실패 / DB 끊김 시 fallback

---

## 8. 모호 자가 체크 (명세 작성 직후)

명세서 emit 직전 본문에 다음 단어가 있는지 grep:

```
적당히 | 잘 | 자연스럽게 | 필요 시 | 가끔 | 보통 | 일반적 | 충분히 | 여유롭게 | 비슷하게 | 등 | 같은 | 비교적
```

각 매치마다 "*얼마/언제/어떤 조건*인지 수치·단위로 치환했나?" 확인.

---

## 9. PoC 교육 메시지

명세서 작성 단계는 **사람의 사고를 결정론으로 만드는 작업**입니다. LLM은 빈 영역을 임의로 채우는 *확률적 함수*라, 모호한 명세를 주면 결과도 확률적입니다. 같은 prompt를 두 번 돌렸을 때 다른 결과가 나온다면 *명세에 모호한 영역*이 그만큼 있다는 신호입니다.

PoC의 핵심은 **"명세 → 결정론적 출력"** 의 메커니즘 시연. 그 첫걸음은 *명세 자체를 결정론적으로 쓰는 것*입니다.

## 적용 흐름 (제안)

1. 새 명세서 작성 전 — 본 문서를 한 번 통독.
2. 작성 중 — §1~§7 체크리스트로 항목별 검토.
3. 작성 직후 — §8 모호 자가 체크.
4. 가능하면 향후 **SpecLint Agent** (Phase 0.5) 자동화로 본 체크리스트 위반을 LLM/정적 분석으로 잡기.
