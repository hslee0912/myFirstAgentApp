# 판타지 슈팅 게임 프로젝트 — 전체 구성 지시서

> **§0. 게임 메타 정보**
>
> - **장르**: 판타지 종스크롤 슈팅 (single-player, 키보드 조작)
> - **레퍼런스**: 마성전설(Knightmare) 오마주. Demo는 React Canvas로 구현, 향후 Phaser.js 리팩토링 예정 (이번 단계 X).
> - **best score 의미**: **Yes** — `/api/v1/game/best` endpoint 사용.
> - **장기 상태 변수**: HP, MP, 현재 stage, score
> - **플레이 시간**: 약 3~10분 (HP 0 도달 시 종료)
> - **저장 단위**: 한 판 끝나면 결과 1행 INSERT

---

## 1. 프로젝트 개요

- 마성전설(Knightmare) 오마주 판타지 종스크롤 슈팅 게임.
- 주인공이 무기를 골라 적을 처치하면서 스테이지를 진행. HP/MP 자원 관리 + 무기 패턴 선택이 핵심.
- Demo는 React Canvas로 1000×750 화면에서 진행.

## 4. MySQL 스키마 (테이블 2개 + 시드)

`BE/db/migrations/<UTC ts>_<name>.sql` emit. orchestrator Phase 2.5가 자동 적용. **charset/collation은 `utf8mb4` / `utf8mb4_unicode_ci`** (한글 player_name). `CREATE INDEX IF NOT EXISTS`는 MySQL 미지원 — INDEX는 `CREATE TABLE` 안에 선언.

### 4-1. `player_users` (모든 게임 완전 고정 — 수정 금지)

| 컬럼 | 타입 | 비고 |
|---|---|---|
| id | BIGINT PK AUTO_INCREMENT | |
| username | VARCHAR(32) UNIQUE NOT NULL | 영문/숫자/언더스코어, 4~16자 |
| password_hash | CHAR(60) NOT NULL | bcrypt rounds=10 (60자) |
| player_name | VARCHAR(60) NOT NULL | 한글 허용, 2~12자 |
| created_at | DATETIME DEFAULT CURRENT_TIMESTAMP | |

### 4-2. `game_results`

**고정 컬럼** (모든 게임 공통):

| 컬럼 | 타입 | 비고 |
|---|---|---|
| id | BIGINT PK AUTO_INCREMENT | |
| player_id | BIGINT NOT NULL | FK → player_users(id) |
| score | INT NOT NULL | |
| play_time | DATETIME NOT NULL | MySQL `'YYYY-MM-DD HH:MM:SS'` 형식만. FE가 ISO 8601 보내면 BE에서 `new Date(s).toISOString().slice(0,19).replace('T',' ')`로 변환 |
| is_best | TINYINT(1) DEFAULT 0 | 개인 최고 여부 |
| INDEX | (player_id, score DESC) | |

**이 게임 추가 컬럼**:

| 컬럼 | 타입 | 비고 |
|---|---|---|
| stage | TINYINT NOT NULL | 1~5 |
| weapon_used | VARCHAR(32) NOT NULL | `holy_sword` / `thunder_lance` / `fire_magic` / `holy_light` |
| play_duration | INT NOT NULL | 초 |
| pattern_seed | VARCHAR(64) | nullable |

### 4-3. 시드 (INSERT IGNORE 1건씩)

- `player_users`: `(username='demo_user', password_hash='$2b$10$fmKqy9chmRPcdIq6nEbRoeP7bNX7CPQmZ7felu0VOleDBqeUO96Aq', player_name='데모용사')`
  - ⚠️ password_hash는 *위 60자 그대로* — `bcrypt.compareSync('Pass1234', hash) === true` 검증된 실제 hash. 다른 hash로 교체 시 login valid_credentials 시나리오 401 FAIL.
- `game_results`: `(player_id=1, stage=1, score=500, weapon_used='holy_sword', play_duration=120, pattern_seed='seed_001', is_best=1, play_time=NOW())`

⚠️ **시드 정합성** (PostTest fail 방지) — 시드 row가 모든 BE validator를 통과해야 함:
- username regex가 underscore 허용해야 시드 `demo_user` 통과 (§5-5-1 #2).
- weapon_used enum에 `holy_sword` 포함.
- stage 1~5 범위에 1 포함.

## 5. 회원가입 / 로그인 규칙 (모든 게임 동일 — rules/domain.md §2 참조)

### 5-1. 입력 필드 (`MemberShipPage.jsx`)

| 필드 | FE 검증 (실시간) | BE 재검증 |
|---|---|---|
| ID (`username`) | `^[a-zA-Z0-9_]{4,16}$`, **/auth/check로 중복 확인 필수** | 정규식 + UNIQUE |
| Password | 8자 이상, 영문+숫자 혼합 (`^(?=.*[A-Za-z])(?=.*\d)[A-Za-z\d]{8,}$`) | 동일 정규식 |
| Confirm Password | Password와 완전 일치해야 가입 버튼 활성 | (FE only) |
| Player Name | 한글/영문/숫자, 2~12자 | 길이 검증 |

### 5-2. 가입 버튼 활성 조건 (FE, AND)
1. 모든 필드 클라이언트 검증 PASS
2. Confirm Password = Password
3. **`/auth/check` 응답 `exists:false`로 중복확인 통과**

ID 입력이 바뀌면 *중복확인 상태 reset* — 다시 누르기 전까진 가입 비활성.

### 5-3. 보안

- 평문 저장 금지. **`bcrypt`** (`bcryptjs` 아님) `bcrypt.hash(password, 10)`.
- 로그인 응답에 토큰 X. `player_id` / `username` / `player_name`만 — FE가 sessionStorage/state에 보관.

### 5-4. BE Endpoint 5종 — contract와 정확히 일치 (D66 가드)

| # | Endpoint | Method | 핵심 |
|---|---|---|---|
| 1 | `/api/v1/auth/signup` | POST | 회원가입 + 중복 체크 + bcrypt 해시 |
| 2 | `/api/v1/auth/check` | GET | username 중복 여부 |
| 3 | `/api/v1/auth/login` | POST | bcrypt.compare + 세션 정보 |
| 4 | `/api/v1/game/result` | POST | 결과 저장 + is_best 계산 |
| 5 | `/api/v1/game/best` | GET | 개인 최고 + 이력 없으면 default |

> ⚠️ FE는 이 목록의 path만 fetch. 다른 path 발명 시 `FE_CONTRACT_DRIFT` ERROR. BE는 5개 모두 mount해야 ContractSync PASS.

#### 5-5-1. `POST /api/v1/auth/signup` (반드시 이 순서)

1. body `{username, password, player_name}` 누락 → `400 {success:false, error:"Missing required fields"}`
2. `validateUsername(username)` 실패 → `400 {error:"Invalid username format"}`
3. `validatePassword(password)` 실패 → `400 {error:"Password too short or invalid"}`
4. `validatePlayerName(player_name)` 실패 → `400 {error:"Invalid player name"}`
5. `SELECT id FROM player_users WHERE username=?` → 있으면 `409 {error:"Username already exists"}`
6. `bcrypt.hash(password, 10)`
7. `INSERT INTO player_users (username, password_hash, player_name) VALUES (?, ?, ?)`
8. `201 {success:true, data:{id:insertId, username, player_name}}`

#### 5-5-2. `GET /api/v1/auth/check?username=<string>`

1. query.username 누락/형식 위반 → `400 {error:"Invalid username"}`
2. `SELECT id FROM player_users WHERE username=?`
3. `200 {success:true, data:{exists:boolean}}`

#### 5-5-3. `POST /api/v1/auth/login`

1. body `{username, password}` 누락 → `400`
2. `SELECT id, username, password_hash, player_name FROM player_users WHERE username=?`
3. row 없으면 → `401 {error:"Invalid username or password"}` (계정 존재 누설 방지)
4. `bcrypt.compare(password, row.password_hash)` false → 같은 `401`
5. `200 {success:true, data:{player_id:row.id, username, player_name}}`

#### 5-5-4. `POST /api/v1/game/result` (⚠️ datetime 변환 필수)

1. body `{player_id, stage, score, weapon_used, play_duration, pattern_seed, play_time}` 누락 → `400`
2. `validateStage(stage)` (1~5) / `validateNonNegativeInt(score)` / `validateNonNegativeInt(play_duration)` / weapon_used ∈ enum 검증 → 위반 `400`
3. **`play_time` 변환**: `new Date(play_time).toISOString().slice(0,19).replace('T',' ')` — 안 하면 `ER_TRUNCATED_WRONG_VALUE` 500
4. `SELECT MAX(score) FROM game_results WHERE player_id=?` (없으면 0)
5. `score > currentBest`면 `is_best=1` + 같은 player 기존 `is_best=1` row를 `UPDATE ... SET is_best=0`
6. `INSERT INTO game_results (...) VALUES (...)`
7. `201 {success:true, data:{result_id:insertId, is_best}}`

#### 5-5-5. `GET /api/v1/game/best?player_id=<integer>` (⚠️ default 필수)

1. query.player_id 누락/숫자 아님 → `400`
2. `SELECT score, stage, weapon_used FROM game_results WHERE player_id=? AND is_best=1 ORDER BY score DESC LIMIT 1`
3. row 없으면 → `200 {success:true, data:{best_score:0, stage_reached:0, weapon_used:'none'}}` *(null 반환 금지 — contract type 유지)*
4. row 있으면 → `200 {success:true, data:{best_score:row.score, stage_reached:row.stage, weapon_used:row.weapon_used}}`

> ⚠️ **PostTest 시나리오 안정성** — `game/result` 시나리오 score는 *모두 500 미만* (예: 100, 200). `game/best` 시나리오는 시드 그대로 검증 (`{data:{best_score:500, stage_reached:1, weapon_used:"holy_sword"}}`). 집계 endpoint(best/top/latest)는 시드 검증, 그 시드를 갈아치울 다른 endpoint 시나리오값은 시드보다 작게.

### 5-6. 중복확인 FE 흐름

1. ID 입력 → 클라이언트 검증 PASS → "중복확인" 버튼 enabled
2. 클릭 → `GET /api/v1/auth/check?username=<encoded>`
3. `exists:false` → 사용 가능 + `usernameAvailable=true`
4. `exists:true` → 이미 사용 중 + `usernameAvailable=false`
5. ID 변경 시 `usernameChecked=false` reset

## 7. 게임 규칙

### 7-1. 장르·분류

판타지 종스크롤 슈팅. 4종 무기 선택 + 5단계 스테이지 + 한 발 즉사 적 + 시각적 난이도 곡선.

### 7-2. 시각 형태 — 적 모양·색 1:1 매핑 (필수)

- 캔버스 1000×750
- **주인공**: 흰색 원
- **적**: 4종 도형 × 4종 무기 색. **적이 들고 있는 무기에 따라 *모양*과 *색*이 1:1 고정** (한 적이 다른 무기를 도중에 바꾸지 않음). 매핑 표:

| 적이 든 무기 | 모양 | Canvas 그리기 | 색 (무기 색 그대로) |
|---|---|---|---|
| `holy_sword` | 세모 (▲ 정삼각형, 꼭짓점이 위) | `moveTo + lineTo` 3점 | `#FFD700` |
| `thunder_lance` | 별 (★ 5각 별) | 10점(외/내) lineTo | `#A347FF` |
| `fire_magic` | 네모 (■ 정사각형) | `fillRect` 또는 4점 path | `#FF3B30` |
| `holy_light` | 역삼각형 (▽ 꼭짓점이 아래) | `moveTo + lineTo` 3점 | `#33E0E0` |

위 표 순서는 §7-4 무기 표 순서와 1:1 일치 (LLM이 매핑을 헷갈리지 않도록 명세서 전체에서 *동일 순서* 유지).

- **탄환**: 무기 색 HEX 그대로 (작은 원, 반지름 4px).
- **배경**: stage별 HEX (§7-5).

🚫 **금지**: 모든 적을 동일 모양(예: 모두 세모)으로 그리는 구현. 위 4종 모양이 각각 등장해야 함.

### 7-3. 초기 상태 변수

| 변수 | 초기값 | 범위 | 단위 | clamp |
|---|---|---|---|---|
| HP | 100 | [0, 100] | - | `Math.max(0, Math.min(hp, 100))` |
| MP | 100 | [0, 100] | - | `Math.min(mp+N, 100)` |
| score | 0 | [0, ∞) | - | - |
| currentStage | 1 | [1, 5] | - | - |

### 7-4. 자원·점수 변화 룰

| 변수 | 증가 트리거 | 변화량 | 감소 트리거 | 감소량 | 주기 | cooldown / 예외 |
|---|---|---|---|---|---|---|
| HP | (회복 없음) | - | 적 탄환 피탄, 적 본체 충돌 | -10 | 즉시 | 탄환·충돌이 아닌 경우 절대 HP 차감 X |
| MP | 자동 회복 | +2 | 공격 (무기별 cost) | -<cost> | 500ms마다 | **공격 직후 300ms 동안 회복 X** (cooldown). 일시정지(ESC) 중 회복 정지. |
| score | 적 격파 | §7-8 stage별 | - | - | 즉시 | - |
| currentStage | score threshold 도달 | +1 | (감소 없음) | - | 즉시 | §7-5 stage별 threshold |

**주인공 무기 4종 — cost / 패턴 / 색 / 발사 딜레이**:

| 무기 | id | cost | 발사 패턴 | 색상 HEX | 발사 딜레이 (ms) |
|---|---|---|---|---|---|
| 성검 | `holy_sword` | 0 (무제한) | 직선 단발 | `#FFD700` | 100 |
| 번개 창 | `thunder_lance` | 3 | 관통 고속 직선 (`pierce:true`) | `#A347FF` | 300 |
| 불꽃 마법 | `fire_magic` | 5 | 부채꼴 3발 ±20° | `#FF3B30` | 200 |
| 성스러운 빛 | `holy_light` | 8 | 360° 8방향 단발 | `#33E0E0` | 400 |

- `mp < cost`이면 발사 *불가*. 성검은 cost 0이라 MP 무관 발사 가능.
- **발사 딜레이** = 같은 무기 *연속 발사 시 최소 간격*. 마지막 발사 시각을 `lastFireTime[weaponId]`에 기록 → `now - lastFireTime[weaponId] >= fireDelayMs[weaponId]` 일 때만 발사 허용. Space 키를 빠르게 연타해도 표 ms 이내엔 발사 X. 무기 슬롯 전환 시 *각 무기 cooldown은 개별 유지* (성검 발사 후 0.3초 뒤 번개로 전환해도 번개의 cooldown은 별개).
- 4색은 서로 다르고 주인공 흰색과 안 겹침.

### 7-5. 진행 단위 — 5 스테이지

| Stage | 테마 | 전환 조건 (누적 score) | **배경색 HEX (필수)** |
|---|---|---|---|
| 1 | 마법의 숲 | 누적 score ≥ 1000 → Stage 2 | `#1B4D2E` (짙은 숲 녹색) |
| 2 | 저주의 늪 | 누적 score ≥ 2000 → Stage 3 | `#1A3D3A` (어두운 청록 늪) |
| 3 | 용암 화산 | 누적 score ≥ 3000 → Stage 4 | `#5C1A0E` (용암 진홍/적갈) |
| 4 | 암흑 성채 | 누적 score ≥ 4000 → Stage 5 | `#1E1A33` (어둠 자주/회보라) |
| 5 | 마왕의 탑 | 최종, 전환 없음 | `#0A0A0A` (칠흑) |

- Stage 선택 UI 없이 **무조건 1부터 시작**. 매 frame `fillRect(0,0,1000,750)`로 전체 배경 칠하기.
- 배경색 HEX는 *위 표 값 그대로* 사용 (LLM 임의 변경 금지 — 시각 회귀).

🚫 **금지 패턴**: 모든 stage threshold를 1000으로 동일하게 두면 score=1000 도달 시 매 frame `stage++` 반복 → 1 frame 안에 stage 1→5 점프. 반드시 *누적 score step 1000*씩 증가하는 threshold.

### 7-6. 게임오버 조건 (frame 단위 명시)

`hp <= 0`이 되는 *그 frame*에:

1. 같은 frame 안에 `navigate('/result')` 호출. setTimeout/setInterval 사용 금지 (지연 발생).
2. game loop 즉시 중단 (`cancelAnimationFrame` + state flag `gameOver=true`).
3. 진행 중인 적 spawn / 탄환 update / MP 회복 모두 중단 (gameOver flag 체크로 skip).
4. ResultPage 진입 시 마지막 frame state (score / stage / weapon_used / play_duration) 전달.

🚫 **금지 패턴**:
- HP가 *음수로 한참 떨어진 후* 게임오버 (frame 누적 update 지연).
- 게임오버 직후에도 적이 한두 frame 더 spawn.
- HP=0인데 탄환 update 한 frame 더 진행되어 score 추가 가산.

#### 7-9-2. spawn 보충 룰 (필수 — 적이 없으면 게임 진행 불가)

게임 진행 동안 *화면에 적이 끊기지 않도록* 자동 spawn:

| 항목 | 값 |
|---|---|
| 최소 화면 적 수 | **5마리** (이보다 적으면 즉시 다음 spawn 시각 앞당김) |
| 최대 화면 적 수 | **12마리** (이보다 많으면 spawn 일시 중단) |
| 정상 spawn 주기 | Stage 1: 1500ms / Stage 2: 1300ms / Stage 3: 1100ms / Stage 4: 900ms / Stage 5: 700ms (단계적 난이도 ↑) |
| 죽은 적 보충 | 한 마리 사망 시 *그 다음 spawn tick에서* pool 중 1마리 자동 등장 (별도 timer 추가 불필요 — 정상 주기가 알아서 채움) |
| 첫 spawn | 게임 시작 후 500ms 유예 뒤 첫 적 등장 |

🚫 **금지 패턴**:
- 한 번 죽으면 다시 spawn 안 함 (게임 진행 불가)
- spawn 주기 *고정 1000ms* 같은 stage 무관 단일값 (난이도 평탄화)
- 화면 적 0마리인 상태가 *2초 이상* 유지

#### 7-9-3. 이동 패턴 (스폰 시 3종 중 1개 무작위 부여)

1. **직선 하강**: 화면 상단에서 일정 속도로 아래로 (속도 80 px/s).
2. **지그재그 하강**: 좌우 sin 곡선 (진폭 80px, 주파수 1Hz, 평균 하강속도 70 px/s).
3. **타원 편대 하강**: 3마리 줄지어 타원 궤적 진입 후 하강 (평균 70 px/s).

화면 하단(y > 750)에 닿으면 *사라짐* (점수 가산 X). gameOver 시 update 중단.

#### 7-9-4. 공격 패턴 — *§7-4 주인공 무기와 동일* (튜닝 금지)

적은 자기 무기로 주인공을 향해 발사. **탄환 패턴(탄수·각도·상대 속도)은 주인공 무기와 완전 동일** — 차이는 *발사 주기*와 *진행 방향*뿐.

| 적 | 무기 (패턴) | 발사 주기 (ms) | 탄환 속성 |
|---|---|---|---|
| 성검 적 (세모) | `holy_sword` (직선 단발) | 1200 | 일반 |
| 번개 적 (별) | `thunder_lance` (관통 고속 aimed) | 1500 | **`pierce:true` 필수** |
| 불꽃 적 (네모) | `fire_magic` (부채꼴 3발 ±20°) | 2000 | 일반 |
| 성광 적 (역삼각형) | `holy_light` (360° 8방향) | 3000 | 일반 |

**필수**:
1. 발사 주기는 표 ms 값 *그대로* (`cooldown * 1.5` 같은 임의 배수 금지). *적 발사 주기*와 *주인공 발사 딜레이*(§7-4)는 서로 다른 개념 — 적은 자기 마릿당 표 주기로 자동 발사, 주인공은 Space 키 입력 + §7-4 딜레이.
2. 번개 적탄 `pierce:true` — 플레이어 충돌 시 HP만 깎고 탄환 *유지*. 일반 탄환은 소멸.
3. 탄환 색 = §7-4 무기 색 HEX 그대로.
4. 스폰 후 **0.5초 유예** 뒤 첫 발사 (`spawnTime + 500ms`).
5. 적 발사 분기 라벨은 주인공 라벨과 동일 (`single/fan/pierce/omni`) — *플레이어/적 공통 helper* 권장.

**조준 방식 (필수)**:

발사 *시점*의 주인공 위치를 기준으로 **직선 발사**. 탄환은 등록 시점에 `dx`, `dy`를 계산하고 매 frame 그 값으로 위치 update. **발사 이후 dx/dy 변경 금지**.

```js
// ✅ 좋은 예 — 발사 시점 계산, 이후 불변
function spawnEnemyBullet(enemy, player) {
  const dx = player.x - enemy.x;  // 발사 시점 1회만
  const dy = player.y - enemy.y;
  const len = Math.hypot(dx, dy);
  const speed = 250;  // px/s
  return { x: enemy.x, y: enemy.y, vx: (dx/len)*speed, vy: (dy/len)*speed, pierce: enemy.weapon === 'thunder_lance' };
}
// frame update: bullet.x += bullet.vx * dt; bullet.y += bullet.vy * dt;  // vx/vy 재계산 X
```

🚫 **금지 패턴** (LLM이 흔히 잘못 구현):
- **유도탄 (homing)**: 매 frame `dx = player.x - bullet.x` 재계산. 절대 금지.
- **추적 (tracking)**: setInterval로 주기적 vx/vy 갱신. 금지.
- **곡선 궤적**: sin/cos 적용한 휘어진 경로. 금지 (단 `holy_light` 8방향 단발은 *각도 8개 고정* 직선 OK).
- **속도 임의 변형**: 표 외 속도 (예: 400 px/s 같은 임의 값). 금지.

#### 7-9-5. 피격·사망 — *한 발 즉사*

적은 **플레이어 탄환에 한 발이라도 맞으면 즉시 사망**. 적은 HP 개념 *없음*. `bullet ↔ enemy` 충돌 감지 시:

1. 해당 적을 즉시 제거 (`enemy.alive = false` 또는 enemies 배열에서 splice)
2. 탄환도 제거 (단 *주인공* `pierce:true` 탄환은 예외 — §7-4. 적의 `pierce`는 *주인공* 피격에만 적용, 적 자신은 *항상 즉사*)
3. score 가산 (§7-8 stage별)
4. §7-9-2 spawn 보충 룰에 따라 *다음 spawn tick*에서 자동 충원

🚫 **금지 사항**:
- 적에 hp/health/durability 필드.
- 여러 발 누적 데미지.
- 적 무적 시간.
- "강한 적이라서" 즉사 룰 예외.
