## 4. MySQL 스키마 (테이블 2개 + 시드)

> `BE/db/migrations/<UTC ts>_<name>.sql` emit. orchestrator Phase 2.5가 자동 적용. **charset/collation은 `utf8mb4` / `utf8mb4_unicode_ci`** (한글 player_name).
> `CREATE INDEX IF NOT EXISTS`는 MySQL 미지원 — INDEX는 `CREATE TABLE` 안에 선언.

### 4-1. `player_users`
| 컬럼 | 타입 | 비고 |
|---|---|---|
| id | BIGINT PK AUTO_INCREMENT | |
| username | VARCHAR(32) UNIQUE NOT NULL | 영문/숫자/언더스코어, 4~16자 |
| password_hash | CHAR(60) NOT NULL | bcrypt rounds=10 (60자) |
| player_name | VARCHAR(60) NOT NULL | 한글 허용, 2~12자 |
| created_at | DATETIME DEFAULT CURRENT_TIMESTAMP | |

### 4-2. `game_results`
| 컬럼 | 타입 | 비고 |
|---|---|---|
| id | BIGINT PK AUTO_INCREMENT | |
| player_id | BIGINT NOT NULL | FK → player_users(id) |
| stage | TINYINT NOT NULL | 1~5 |
| score | INT NOT NULL | |
| weapon_used | VARCHAR(32) NOT NULL | holy_sword / fire_magic / thunder_lance / holy_light |
| play_duration | INT NOT NULL | 초 |
| pattern_seed | VARCHAR(64) | nullable |
| is_best | TINYINT(1) DEFAULT 0 | 개인 최고 점수 여부 |
| play_time | DATETIME NOT NULL | ⚠️ MySQL `'YYYY-MM-DD HH:MM:SS'` 형식만. FE가 ISO 8601 보내면 BE에서 `new Date(s).toISOString().slice(0,19).replace('T',' ')`로 변환 (안 하면 500). |
| INDEX | (player_id, score DESC) | |

### 4-3. 시드 (INSERT IGNORE 1건씩)

- `player_users`: `(username='demo_user', password_hash='$2b$10$fmKqy9chmRPcdIq6nEbRoeP7bNX7CPQmZ7felu0VOleDBqeUO96Aq', player_name='데모용사')`
- `game_results`: `(player_id=1, stage=1, score=500, weapon_used='holy_sword', play_duration=120, pattern_seed='seed_001', is_best=1, play_time=NOW())`

> ⚠️ **시드 정합성 (PostTest fail 방지)** — 시드 row는 BE 핸들러 validation을 *전부* 통과해야 함:
> - username regex가 underscore 허용해야 시드 `demo_user` 통과 (§5-5-1 #2).
> - `password_hash`는 위 60자 hash *그대로* 박을 것 — `bcrypt.compareSync('Pass1234', hash) === true` 검증 완료된 실제 hash. 예시처럼 보여 placeholder로 오인하고 다른 hash로 교체 시 login valid_credentials 시나리오 401 fail.

## 5. 회원가입 / 로그인 규칙

### 5-1. 입력 필드 (MemberShipPage.jsx)

| 필드 | FE 검증 (실시간) | BE 재검증 |
|---|---|---|
| ID (`username`) | 영문/숫자/언더스코어 4~16자, **/auth/check로 중복 확인 필수** | 정규식 + UNIQUE |
| Password | 8자 이상, 영문+숫자 혼합 | 동일 정규식 |
| Confirm Password | Password와 완전 일치해야 가입 버튼 활성 | (FE only) |
| Player Name | 한글/영문/숫자, 2~12자 | 길이 검증 |

### 5-2. 가입 버튼 활성 조건 (FE, AND)
1. 모든 필드 클라이언트 검증 PASS
2. Confirm Password = Password
3. **`/auth/check` 응답 `exists:false`로 중복확인 통과**

ID 입력이 바뀌면 *중복확인 상태 reset* — 다시 누르기 전까진 가입 비활성.

### 5-3. 보안

- 평문 저장 금지. **`bcrypt`** (정확한 철자, `bcryptjs` 아님) `bcrypt.hash(password, 10)` 으로 해시.
- 로그인 응답에 토큰 X. `player_id` / `username` / `player_name`만 — FE가 sessionStorage/state에 보관.

### 5-4. BE Endpoint 5종 — contract와 정확히 일치 (D66 가드)

| # | Endpoint | Method | 핵심 |
|---|---|---|---|
| 1 | `/api/v1/auth/signup` | POST | 회원가입 + 중복 체크 + bcrypt 해시 |
| 2 | `/api/v1/auth/check` | GET | username 중복 여부 |
| 3 | `/api/v1/auth/login` | POST | bcrypt.compare + 세션 정보 응답 |
| 4 | `/api/v1/game/result` | POST | 결과 저장 + is_best 계산 |
| 5 | `/api/v1/game/best` | GET | 개인 최고 + 이력 없으면 default |

> ⚠️ FE는 이 목록의 path만 fetch. 다른 path 발명 시 `FE_CONTRACT_DRIFT` ERROR. BE는 5개 모두 mount해야 ContractSync PASS.

#### 5-5-1. `POST /api/v1/auth/signup` (반드시 이 순서)

1. body `{username, password, player_name}` 누락 → `400 {success:false, error:"Missing required fields"}`
2. username 정규식 `/^[a-zA-Z0-9_]{4,16}$/` (underscore 포함, 시드 `demo_user` 통과 필수) → 실패 `400 {error:"Invalid username format"}`
3. password 정규식 `8자 이상 + 영문+숫자 혼합` → 실패 `400 {error:"Password too short or invalid"}`
4. player_name 길이 2~12 (`String#length`) → 위반 `400`
5. `SELECT id FROM player_users WHERE username=?` → 있으면 `409 {error:"Username already exists"}`
6. `bcrypt.hash(password, 10)` (정확한 패키지)
7. `INSERT INTO player_users (username, password_hash, player_name) VALUES (?, ?, ?)`
8. `201 {success:true, data:{id:insertId, username, player_name}}`

#### 5-5-2. `GET /api/v1/auth/check?username=<string>`

1. query.username 누락/형식 위반 → `400 {error:"Invalid username"}`
2. `SELECT id FROM player_users WHERE username=?`
3. `200 {success:true, data:{exists:boolean}}`

#### 5-5-3. `POST /api/v1/auth/login`

1. body `{username, password}` 누락 → `400`
2. `SELECT id, username, password_hash, player_name FROM player_users WHERE username=?`
3. row 없으면 → `401 {success:false, error:"Invalid username or password"}` (계정 존재 누설 방지)
4. `bcrypt.compare(password, row.password_hash)` false → 같은 `401`
5. `200 {success:true, data:{player_id:row.id, username, player_name}}`

#### 5-5-4. `POST /api/v1/game/result` (⚠️ datetime 변환 필수)

1. body `{player_id, stage, score, weapon_used, play_duration, pattern_seed, play_time}` 누락 → `400`
2. stage 1~5 / score≥0 / play_duration≥0 / weapon_used ∈ `['holy_sword','fire_magic','thunder_lance','holy_light']` → 위반 `400`
3. **`play_time` 변환** (필수): `new Date(play_time).toISOString().slice(0,19).replace('T',' ')` — 안 하면 `ER_TRUNCATED_WRONG_VALUE` 500
4. `SELECT MAX(score) FROM game_results WHERE player_id=?` (없으면 0)
5. `score > currentBest`면 `is_best=1` + 같은 player 기존 `is_best=1` row를 `UPDATE ... SET is_best=0`
6. `INSERT INTO game_results (player_id, stage, score, weapon_used, play_duration, pattern_seed, is_best, play_time) VALUES (?, ?, ?, ?, ?, ?, ?, ?)`
7. `201 {success:true, data:{result_id:insertId, is_best}}`

#### 5-5-5. `GET /api/v1/game/best?player_id=<integer>` (⚠️ default 필수)

1. query.player_id 누락/숫자 아님 → `400`
2. `SELECT score, stage, weapon_used FROM game_results WHERE player_id=? AND is_best=1 ORDER BY score DESC LIMIT 1`
3. row 없으면 → `200 {success:true, data:{best_score:0, stage_reached:0, weapon_used:'none'}}` *(null 반환 금지 — contract type 유지)*
4. row 있으면 → `200 {success:true, data:{best_score:row.score, stage_reached:row.stage, weapon_used:row.weapon_used}}`

> ⚠️ **PostTest 시나리오 안정성 — 시드 best 보존 (CodeChecker가 test_scenarios 작성 시 *필수*)**
> PostTest는 endpoint × 시나리오를 *동일 컨테이너/DB*에 직렬 실행. `game/result` POST가 시드(500)보다 큰 score를 넣으면 is_best 갱신 → 직후 `game/best` GET 시나리오가 시드 mismatch FAIL.
> **룰:** `game/result` 시나리오 score는 *모두 500 미만* (예: 100, 200). `game/best` 시나리오는 시드 그대로 검증 (`{data:{best_score:500, stage_reached:1, weapon_used:"holy_sword"}}`). 일반 원칙: *집계 endpoint(best/top/latest)는 시드 검증, 그 시드를 갈아치울 다른 endpoint 시나리오값은 시드보다 작게*.

### 5-6. 중복확인 FE 흐름

1. ID 입력 → 클라이언트 검증 PASS → "중복확인" 버튼 enabled
2. 클릭 → `GET /api/v1/auth/check?username=<encoded>`
3. `exists:false` → 사용 가능 메시지 + `usernameAvailable=true`
4. `exists:true` → 이미 사용중 + `usernameAvailable=false`
5. ID 변경 시 `usernameChecked=false` reset

## 8. 스테이지

5개 스테이지의 `stage` 번호·`theme`·`scoreThreshold`·`bgColor` 정의 = `FE/src/constants/game.js`의 `STAGES` 배열 (placeholder).

- Stage 선택 UI 없이 **무조건 1부터 시작**. 스테이지별 배경 그래픽 다르게.

### 8-bis. ⚠️ 스테이지 배경색 — *반드시 stage 제목과 매핑* (시각 검증 핵심)

스테이지가 바뀌었는데 *배경이 안 바뀌면* 사용자는 stage 전환 자체를 인지 못함. 따라서 배경색은 *stage 제목과 1:1 매핑*되어야 하고, *매 frame render의 첫 fillRect*가 *현재 stage 배경색*을 사용해야 함.

**❌ 나쁨** — 배경색이 stage와 무관 / hardcoded 검정:
```js
renderCtx.fillStyle = '#000';
renderCtx.fillRect(0, 0, w, h);
```

**✅ 좋음** — `STAGES` placeholder를 import하여 `bgColor`를 render 첫 줄에서 사용:
```js
import { STAGES, CANVAS } from '../constants/game';

// 매 frame 렌더 첫 줄
const current = STAGES[gs.stage - 1] || STAGES[0];
renderCtx.fillStyle = current.bgColor;   // ← stage 따라 자동 교체
renderCtx.fillRect(0, 0, CANVAS.width, CANVAS.height);
```

**핵심 룰:**
- 배경색 HEX는 *`STAGES[i].bgColor`의 정확한 값을 그대로 사용* (LLM이 임의로 다른 색을 적으면 시각 회귀).
- 캔버스 *전체*를 매 frame `fillRect(0,0,CANVAS.width,CANVAS.height)`로 칠하기 (이전 frame 잔상 + 배경 바뀜 동시 해결).
- HUD/score/HP 텍스트 영역도 같은 배경 위에 그리되, 가독성 위해 *반투명 검정 박스* 위에 텍스트를 올릴 것.
- 스테이지 제목(`STAGES[i].theme`)도 HUD 한 구석에 *짧게* 표시 (예: 좌상단 "Stage 1 · Magic Forest") — 사용자가 현재 stage 인지 가능.

> ⚠️ **stage 전환 버그 패턴** — 모든 스테이지의 threshold를 동일하게 두면 한 score 도달 시 매 frame에서 stage++가 반복 발화 → 1 frame 안에 stage 1→5 점프. `STAGES[i].scoreThreshold`를 *단계적 증가* 값으로 사용 (placeholder 정의 그대로).

## 9. 적(Enemy) 명세

### 9-1. 종류
- **총 4종**. 각 적은 **무기 4종 중 하나와 1:1로 매핑**되어 자기 무기를 들고 등장.
- 스폰 시 무기 결정: `const pool = ENEMY_POOLS[s.currentStage]; const weaponId = pool[Math.floor(Math.random()*pool.length)]` (`ENEMY_POOLS` placeholder).
- **적 모양 = `ENEMY_SHAPE_BY_WEAPON[weaponId]`** (placeholder).
- **적 색깔 = `WEAPONS.find(w => w.id === weaponId).color`** (placeholder의 무기 색 그대로).

### 9-2. 이동 패턴 (스폰 시 아래 3종 중 1개를 무작위 부여)
1. **직선 하강**: 화면 상단에서 일정 속도로 아래로 내려옴.
2. **지그재그 하강**: 좌우로 sin 곡선을 그리며 내려옴.
3. **타원 편대 하강**: 여러 마리가 줄지어 타원형 궤적을 그리며 진입 후 하강.

### 9-3. 공격 패턴 (이번 보완의 핵심)
- 적은 **자기 무기로 주인공을 향해 발사**한다.
- **공격 패턴 = `PATTERN_DEFINITIONS[weapon.pattern]`** (placeholder) — 플레이어와 *완전 동일* (탄환 수·각도 offset·관통·속도 배수). 차이는 *기준 방향*뿐.
  - `single`: 직선 1발 (`bullets: [0]`)
  - `fan`: 부채꼴 3발 (`bullets: [-20, 0, 20]`)
  - `pierce`: 관통 고속 1발 (`bullets: [0]`, `speedMul: 1.6`, `pierce: true`)
  - `omni`: 8방향 단발 (`bullets: [0,45,90,...315]`, `fixedDirection: true`, player 무관 고정)
- ⚠️ **`bullets`는 *숫자 배열* (degrees)** — placeholder 구조 그대로 사용. 객체로 wrap하지 말 것.
- 기준 방향 분기 (LLM 구현 시 *반드시* 다음 로직):
  - 플레이어: 위쪽 (`baseAngle = -Math.PI / 2`)
  - 적: 발사 시점 player 위치 (aimed) — `baseAngle = Math.atan2(player.y - enemy.y, player.x - enemy.x)`
  - omni (`fixedDirection: true`)는 base 무관, *bullets의 deg 자체가 절대 각도*
- **플레이어/적 공통 helper** 권장 — 한 함수로 placeholder lookup 후 탄환 배열 생성:
  ```js
  function spawnBullets(weapon, sourcePos, baseAngle) {
    const def = PATTERN_DEFINITIONS[weapon.pattern];
    const speed = BULLET_SPEED * (def.speedMul || 1);
    const base = def.fixedDirection ? 0 : baseAngle;  // fixedDirection이면 deg가 절대 각도
    return def.bullets.map((deg) => {
      const rad = base + (deg * Math.PI) / 180;
      return {
        x: sourcePos.x,
        y: sourcePos.y,
        vx: Math.cos(rad) * speed,
        vy: Math.sin(rad) * speed,
        color: weapon.color,
        pierce: def.pierce || false,
      };
    });
  }
  // 플레이어 발사: spawnBullets(player.weapon, player.pos, -Math.PI / 2)
  // 적 발사:    spawnBullets(enemy.weapon, enemy.pos, Math.atan2(player.y-enemy.y, player.x-enemy.x))
  ```
  → 플레이어·적이 *같은 함수* 호출, *같은 패턴 정의* 사용 → 적별로 다른 공격 패턴 불가능.
- 발사 주기 (무기별 쿨다운) = `ENEMY_FIRE_COOLDOWN_MS[weaponId]` placeholder (ms 단위).
- **탄환 색 = 적의 무기 색** (`WEAPONS.find(...).color`).
- 적은 화면 상단 진입 후 **0.5초 유예** 뒤부터 발사 시작 (스폰 즉시 피탄 방지).

🚫 **금지**: 적별로 *별도 분기*로 패턴 작성 (예: `if (enemy.weapon === 'fire_magic') { 부채꼴; } else if ...`). 모든 적이 `PATTERN_DEFINITIONS[enemy.weapon.pattern]` 한 곳에서 lookup해야 함.

### 9-4. 스코어
- 적이 격파되면 stage별 점수 가산. 값 = `SCORE_PER_STAGE[s.currentStage]` placeholder.
- 코드: `s.score += SCORE_PER_STAGE[s.currentStage]` (단일값 `+= 10` 금지).

### 11-1. `report.html` 다운로드 (Blob + download attribute)

```jsx
const html = `<!DOCTYPE html><html><body><h1>Result</h1>...</body></html>`;
const blob = new Blob([html], { type: 'text/html' });
const url = URL.createObjectURL(blob);
const a = document.createElement('a');
a.href = url; a.download = 'report.html'; a.click();
URL.revokeObjectURL(url);
```
