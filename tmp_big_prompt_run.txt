## 4. MySQL 스키마 (테이블 2개 + 시드)

> `BE/db/migrations/<UTC ts>_<name>.sql` emit. orchestrator Phase 2.5가 자동 적용. **charset/collation은 `utf8mb4` / `utf8mb4_unicode_ci`** (한글 player_name).
> `CREATE INDEX IF NOT EXISTS`는 MySQL 미지원 — INDEX는 `CREATE TABLE` 안에 선언.

### 4-1. `player_users`
| 컬럼 | 타입 | 비고 |
|---|---|---|
| id | BIGINT PK AUTO_INCREMENT | |
| username | VARCHAR(32) UNIQUE NOT NULL | 영문/숫자/언더스코어, 4~16자 |
| password_hash | CHAR(60) NOT NULL | bcrypt rounds=10 (60자) |
| player_name | VARCHAR(60) NOT NULL | 한글 허용, 2~12자 |
| created_at | DATETIME DEFAULT CURRENT_TIMESTAMP | |

### 4-2. `game_results`
| 컬럼 | 타입 | 비고 |
|---|---|---|
| id | BIGINT PK AUTO_INCREMENT | |
| player_id | BIGINT NOT NULL | FK → player_users(id) |
| stage | TINYINT NOT NULL | 1~5 |
| score | INT NOT NULL | |
| weapon_used | VARCHAR(32) NOT NULL | holy_sword / fire_magic / thunder_lance / holy_light |
| play_duration | INT NOT NULL | 초 |
| pattern_seed | VARCHAR(64) | nullable |
| is_best | TINYINT(1) DEFAULT 0 | 개인 최고 점수 여부 |
| play_time | DATETIME NOT NULL | ⚠️ MySQL `'YYYY-MM-DD HH:MM:SS'` 형식만. FE가 ISO 8601 보내면 BE에서 `new Date(s).toISOString().slice(0,19).replace('T',' ')`로 변환 (안 하면 500). |
| INDEX | (player_id, score DESC) | |

### 4-3. 시드 (INSERT IGNORE 1건씩)

- `player_users`: `(username='demo_user', password_hash='$2b$10$fmKqy9chmRPcdIq6nEbRoeP7bNX7CPQmZ7felu0VOleDBqeUO96Aq', player_name='데모용사')`
- `game_results`: `(player_id=1, stage=1, score=500, weapon_used='holy_sword', play_duration=120, pattern_seed='seed_001', is_best=1, play_time=NOW())`

> ⚠️ **시드 정합성 (PostTest fail 방지)** — 시드 row는 BE 핸들러 validation을 *전부* 통과해야 함:
> - username regex가 underscore 허용해야 시드 `demo_user` 통과 (§5-5-1 #2).
> - `password_hash`는 위 60자 hash *그대로* 박을 것 — `bcrypt.compareSync('Pass1234', hash) === true` 검증 완료된 실제 hash. 예시처럼 보여 placeholder로 오인하고 다른 hash로 교체 시 login valid_credentials 시나리오 401 fail.

## 5. 회원가입 / 로그인 규칙

### 5-1. 입력 필드 (MemberShipPage.jsx)

| 필드 | FE 검증 (실시간) | BE 재검증 |
|---|---|---|
| ID (`username`) | 영문/숫자/언더스코어 4~16자, **/auth/check로 중복 확인 필수** | 정규식 + UNIQUE |
| Password | 8자 이상, 영문+숫자 혼합 | 동일 정규식 |
| Confirm Password | Password와 완전 일치해야 가입 버튼 활성 | (FE only) |
| Player Name | 한글/영문/숫자, 2~12자 | 길이 검증 |

### 5-2. 가입 버튼 활성 조건 (FE, AND)
1. 모든 필드 클라이언트 검증 PASS
2. Confirm Password = Password
3. **`/auth/check` 응답 `exists:false`로 중복확인 통과**

ID 입력이 바뀌면 *중복확인 상태 reset* — 다시 누르기 전까진 가입 비활성.

### 5-3. 보안

- 평문 저장 금지. **`bcrypt`** (정확한 철자, `bcryptjs` 아님) `bcrypt.hash(password, 10)` 으로 해시.
- 로그인 응답에 토큰 X. `player_id` / `username` / `player_name`만 — FE가 sessionStorage/state에 보관.

### 5-4. BE Endpoint 5종 — contract와 정확히 일치 (D66 가드)

| # | Endpoint | Method | 핵심 |
|---|---|---|---|
| 1 | `/api/v1/auth/signup` | POST | 회원가입 + 중복 체크 + bcrypt 해시 |
| 2 | `/api/v1/auth/check` | GET | username 중복 여부 |
| 3 | `/api/v1/auth/login` | POST | bcrypt.compare + 세션 정보 응답 |
| 4 | `/api/v1/game/result` | POST | 결과 저장 + is_best 계산 |
| 5 | `/api/v1/game/best` | GET | 개인 최고 + 이력 없으면 default |

> ⚠️ FE는 이 목록의 path만 fetch. 다른 path 발명 시 `FE_CONTRACT_DRIFT` ERROR. BE는 5개 모두 mount해야 ContractSync PASS.

> 📌 **공통 원칙 (§5-5-* 5개 endpoint 전부)** — 아래 단계는 *행동·순서·조건*만 기술.
> 실제 SQL 쿼리·드라이버 호출은 LLM이 §4 스키마(테이블·컬럼 이름) 보고 *직접 작성*.
> Bind parameter (`?`) 사용 = `rules/be.md` (SQL injection 방지) — 명세에 재기술 안 함.
> 응답 셰입 `{success, data, error}` = `rules/common.md §3` 공통 — 매 step에 재명시 안 함.

#### 5-5-1. `POST /api/v1/auth/signup` — 회원가입

목적: 중복 없는 username으로 새 player 생성. 비밀번호 해시는 BE 책임 (FE는 평문 전송, §5-3).
순서 중요 — *값 검증 → 중복 확인 → 해시(비용 큼) → insert* 로 CPU·동시성 race window 최소화.

1. **필수 필드 검증** — body `{username, password, player_name}` 셋 중 하나라도
   누락·null·빈 문자열이면 `400 {success:false, error:"Missing required fields"}`.
   (여기서 잘라야 아래 정규식 step이 `undefined.match`로 throw 안 남.)
2. **username 형식** — 정규식 `/^[a-zA-Z0-9_]{4,16}$/`. 위반 시
   `400 {error:"Invalid username format"}`.
   ⚠️ underscore *반드시* 허용 — 시드 row `demo_user` (§4-3)가 통과해야 PostTest
   login 시나리오 정상. 영문/숫자만 허용으로 좁히면 시드 자체가 invalid.
3. **password 형식** — 8자 이상 + 영문 1자 이상 + 숫자 1자 이상 동시 만족.
   위반 시 `400 {error:"Password too short or invalid"}`.
   FE도 동일 정규식으로 사전 검증하지만 BE 재검증 필수 — FE 우회 공격 차단.
4. **player_name 길이** — `String#length` 2~12자. 한글은 1글자=1 count (JS 기본).
   위반 시 `400`.
5. **username 중복 확인** — `player_users` 테이블에서 동일 `username` row 존재 여부
   조회. 있으면 `409 {error:"Username already exists"}` 로 즉시 종료.
   해시 *전*에 두는 이유: bcrypt rounds=10은 ~100ms CPU → 이미 존재하는 username에
   매번 해싱하면 낭비 + race window 확대. (DB UNIQUE 제약은 최후 방어선, app-level이 primary.)
6. **비밀번호 해시** — `bcrypt.hash(password, 10)`.
   ⚠️ 패키지 이름 정확히 `bcrypt` (네이티브). `bcryptjs`는 별개 패키지로 금지
   (`rules/be.md §5` + `allowedDeps` 위반). rounds=10 → 60자 hash 문자열.
7. **저장** — `player_users` 테이블에 `(username, password_hash, player_name)` 3개 컬럼 insert.
   `id`는 AUTO_INCREMENT, `created_at`은 DEFAULT CURRENT_TIMESTAMP (§4-1 스키마).
8. **응답** — `201 {success:true, data:{id:<insert된 id>, username, player_name}}`.
   🚫 `password_hash`는 응답 *절대 금지* (정보 누설).

#### 5-5-2. `GET /api/v1/auth/check?username=<string>` — 중복 확인

목적: 가입 버튼 활성 조건 (§5-2 #3) — 가입 *전*에 username 가용성을 미리 알림.

1. **query 검증** — `query.username` 누락 또는 §5-5-1 #2 정규식 위반 시
   `400 {error:"Invalid username"}`. (형식이 invalid면 어차피 가입 불가 → 미리 차단.)
2. **존재 여부 조회** — `player_users` 테이블에서 동일 `username` row 존재 여부.
   값 자체는 안 가져와도 됨 — 존재 여부(boolean)만 필요.
3. **응답** — `200 {success:true, data:{exists:<boolean>}}`.
   ⚠️ 존재해도 200 (404 아님) — "조회 자체는 성공" semantics. FE가 `exists`로 분기.

#### 5-5-3. `POST /api/v1/auth/login` — 로그인

목적: 평문 password를 받아 bcrypt.compare로 해시 검증. 토큰 없이 player 정보만 응답
(`rules/be.md §5` — 이 PoC는 세션 토큰 X, FE가 sessionStorage 보관).

1. **필수 필드** — body `{username, password}` 누락 시 `400`.
2. **계정 조회** — `player_users`에서 `username`으로 row 조회. 필요한 컬럼:
   `id`, `username`, `password_hash`, `player_name` (응답 구성용 + 해시 비교용).
3. **계정 없음** — row 없으면 `401 {success:false, error:"Invalid username or password"}`.
   ⚠️ "username not found"라고 *구체적으로 응답하지 말 것* — 계정 존재 여부 누설은
   enumeration 공격 (브루트포스 username 추측) 단서. 비밀번호 오류와 *동일 메시지*.
4. **해시 비교** — `bcrypt.compare(password, row.password_hash)` 가 `false`면
   같은 `401` (위와 동일 메시지). `true`면 다음 step.
5. **응답** — `200 {success:true, data:{player_id:<row.id>, username, player_name}}`.
   🚫 `password_hash`는 응답 *절대 금지*.

#### 5-5-4. `POST /api/v1/game/result` — 결과 저장 + 개인 최고 갱신 (⚠️ 가장 복잡)

목적: 게임 종료 시 점수 row 저장 + 개인 최고(`is_best=1`) 자동 갱신.
*같은 player의 is_best=1 row는 항상 1개* 라는 불변식을 BE가 책임.

1. **필수 필드** — body `{player_id, stage, score, weapon_used, play_duration, pattern_seed, play_time}`
   누락 시 `400`. (`pattern_seed`는 nullable 가능, 다른 필드는 필수.)
2. **값 검증** — 위반 시 `400`:
   - `stage`: 1~5 정수
   - `score`: 0 이상 정수
   - `play_duration`: 0 이상 정수 (초)
   - `weapon_used`: `['holy_sword','fire_magic','thunder_lance','holy_light']` 중 하나
     (이 4종은 `FE/src/constants/game.js`의 `WEAPONS[].id`와 일치해야 함, §7)
3. **`play_time` 변환** ⚠️ 필수 — FE는 ISO 8601 (`2026-05-21T14:30:00.000Z`) 전송,
   MySQL은 `'YYYY-MM-DD HH:MM:SS'` 형식만 수용. 변환식:
   `new Date(play_time).toISOString().slice(0,19).replace('T',' ')`.
   누락 시 MySQL이 `ER_TRUNCATED_WRONG_VALUE` (1292) 500 에러.
4. **현재 최고 점수 조회** — `game_results`에서 해당 `player_id`의 최고 `score` 조회
   (`is_best=1` row의 score, 또는 MAX(score)). row 없으면 0으로 취급.
5. **`is_best` 결정 + 기존 best demotion** — 새 score가 현재 최고보다 *크면*:
   - 새 row의 `is_best=1`로 표시
   - 같은 player의 기존 `is_best=1` row(들)을 모두 `is_best=0`으로 update (불변식 유지)
   새 score가 현재 최고 *이하*면 `is_best=0`. ⚠️ demotion → insert 순서 중요 (반대로
   하면 새 row가 1로 들어간 직후 demotion이 새 row까지 0으로 만들어 best 없어짐).
   더 안전한 패턴: *트랜잭션으로 묶음* (demotion + insert 원자적).
6. **저장** — `game_results`에 8개 컬럼 insert: `(player_id, stage, score, weapon_used,
   play_duration, pattern_seed, is_best, play_time)`. `id`는 AUTO_INCREMENT (§4-2).
7. **응답** — `201 {success:true, data:{result_id:<insert된 id>, is_best:<true|false>}}`.

#### 5-5-5. `GET /api/v1/game/best?player_id=<integer>` — 개인 최고 조회 (⚠️ default 필수)

목적: ResultPage가 "이전 최고와 비교" 표시할 때 호출. *이력 없는 신규 player도 호출* →
404 대신 *default 객체* 반환해서 FE 분기 최소화 (contract type 유지).

1. **query 검증** — `query.player_id` 누락 또는 정수 변환 실패(`Number.isFinite`) 시 `400`.
2. **best row 조회** — `game_results`에서 해당 `player_id` + `is_best=1` 조건의
   row 1개 (이론상 1개 이하, §5-5-4 #5 불변식). 필요 컬럼: `score`, `stage`, `weapon_used`.
3. **이력 없음** — row 없으면 `200 {success:true, data:{best_score:0, stage_reached:0, weapon_used:'none'}}`.
   ⚠️ `data:null` 또는 404 *금지* — `shared/api_contract.json`이 data shape를 *고정*해
   FE 코드가 분기 없이 `data.best_score`를 읽도록 함. null 반환 시 FE에서 throw.
4. **이력 있음** — `200 {success:true, data:{best_score:<score>, stage_reached:<stage>, weapon_used:<weapon_used>}}`.

> ⚠️ **PostTest 시나리오 안정성 — 시드 best 보존 (CodeChecker가 test_scenarios 작성 시 *필수*)**
> PostTest는 endpoint × 시나리오를 *동일 컨테이너/DB*에 직렬 실행. `game/result` POST가 시드(500)보다 큰 score를 넣으면 is_best 갱신 → 직후 `game/best` GET 시나리오가 시드 mismatch FAIL.
> **룰:** `game/result` 시나리오 score는 *모두 500 미만* (예: 100, 200). `game/best` 시나리오는 시드 그대로 검증 (`{data:{best_score:500, stage_reached:1, weapon_used:"holy_sword"}}`). 일반 원칙: *집계 endpoint(best/top/latest)는 시드 검증, 그 시드를 갈아치울 다른 endpoint 시나리오값은 시드보다 작게*.

### 5-6. 중복확인 FE 흐름

1. ID 입력 → 클라이언트 검증 PASS → "중복확인" 버튼 enabled
2. 클릭 → `GET /api/v1/auth/check?username=<encoded>`
3. `exists:false` → 사용 가능 메시지 + `usernameAvailable=true`
4. `exists:true` → 이미 사용중 + `usernameAvailable=false`
5. ID 변경 시 `usernameChecked=false` reset

## 8. 스테이지

5개 스테이지의 `stage` 번호·`theme`·`scoreThreshold`·`bgColor` 정의 = `FE/src/constants/game.js`의 `STAGES` 배열 (placeholder).

- Stage 선택 UI 없이 **무조건 1부터 시작**. 스테이지별 배경 그래픽 다르게.

### 8-bis. ⚠️ 스테이지 배경색 — *반드시 stage 제목과 매핑* (시각 검증 핵심)

스테이지가 바뀌었는데 *배경이 안 바뀌면* 사용자는 stage 전환 자체를 인지 못함. 따라서 배경색은 *stage 제목과 1:1 매핑*되어야 하고, *매 frame render의 첫 fillRect*가 *현재 stage 배경색*을 사용해야 함.

**❌ 나쁨** — 배경색이 stage와 무관 / hardcoded 검정:
```js
renderCtx.fillStyle = '#000';
renderCtx.fillRect(0, 0, w, h);
```

**✅ 좋음** — `STAGES` placeholder를 import하여 `bgColor`를 render 첫 줄에서 사용:
```js
import { STAGES, CANVAS } from '../constants/game';

// 매 frame 렌더 첫 줄
const current = STAGES[gs.stage - 1] || STAGES[0];
renderCtx.fillStyle = current.bgColor;   // ← stage 따라 자동 교체
renderCtx.fillRect(0, 0, CANVAS.width, CANVAS.height);
```

**핵심 룰:**
- 배경색 HEX는 *`STAGES[i].bgColor`의 정확한 값을 그대로 사용* (LLM이 임의로 다른 색을 적으면 시각 회귀).
- 캔버스 *전체*를 매 frame `fillRect(0,0,CANVAS.width,CANVAS.height)`로 칠하기 (이전 frame 잔상 + 배경 바뀜 동시 해결).
- HUD/score/HP 텍스트 영역도 같은 배경 위에 그리되, 가독성 위해 *반투명 검정 박스* 위에 텍스트를 올릴 것.
- 스테이지 제목(`STAGES[i].theme`)도 HUD 한 구석에 *짧게* 표시 (예: 좌상단 "Stage 1 · Magic Forest") — 사용자가 현재 stage 인지 가능.

> ⚠️ **stage 전환 버그 패턴** — 모든 스테이지의 threshold를 동일하게 두면 한 score 도달 시 매 frame에서 stage++가 반복 발화 → 1 frame 안에 stage 1→5 점프. `STAGES[i].scoreThreshold`를 *단계적 증가* 값으로 사용 (placeholder 정의 그대로).

## 9. 적(Enemy) 명세

### 9-1. 종류·외형 (drawEnemy 4-shape 분기 필수)

- **총 4종**. 각 적은 **무기 4종 중 하나와 1:1로 매핑**되어 자기 무기를 들고 등장.
- 스폰 시 무기 결정: `ENEMY_POOLS[currentStage]`에서 무작위 1개 선택 → `weaponId`.
- **외형 lookup** (스폰 시):
  - `shape = ENEMY_SHAPE_BY_WEAPON[weaponId]` (4종 문자열 중 하나, placeholder)
  - `color = WEAPONS.find(w => w.id === weaponId).color` (무기 색 그대로)
  - enemy 객체에 `{weaponId, shape, color, …}` 저장 → 매 frame draw에서 재활용.

**🚫 핵심 룰 — `drawEnemy`는 `enemy.shape` 문자열로 4-way switch 필수**:

placeholder가 4가지 모양 *문자열*을 지정 (`'triangle' | 'star' | 'square' | 'inverted_triangle'`).
단일 path 하드코딩 시 *모든 적이 같은 모양*으로 그려져 무기 종류 시각 구분 불가.

```js
function drawEnemy(ctx, enemy) {
  ctx.save();
  ctx.translate(enemy.x, enemy.y);
  ctx.fillStyle = enemy.color;
  switch (enemy.shape) {
    case 'triangle':          // ▲ 위 꼭짓점, 아래 좌·우
      // beginPath → moveTo(0,-h) → lineTo(w, h) → lineTo(-w, h) → closePath → fill
      break;
    case 'inverted_triangle': // ▽ 아래 꼭짓점, 위 좌·우 (triangle을 위·아래 뒤집기)
      // beginPath → moveTo(0, h) → lineTo(w,-h) → lineTo(-w,-h) → closePath → fill
      break;
    case 'square':            // ■ 정사각형
      // fillRect(-w/2, -w/2, w, w) 또는 path로 직사각형
      break;
    case 'star':              // ★ 5각 별 — 외접원/내접원 반지름 교대로 10점 path
      // for i in 0..9: angle = -PI/2 + i*PI/5; r = (i%2===0) ? rOuter : rInner;
      // beginPath → moveTo/lineTo → closePath → fill
      break;
  }
  ctx.restore();
}
```

크기 가이드: 외접 반지름 ~14px (적이 너무 크면 화면 점령, 너무 작으면 시각 식별 어려움).

### 9-1-bis. 스폰 빈도 — `SPAWN_CONFIG` 필드명 정확히

placeholder의 *정확한 필드명*만 사용. LLM이 `intervalMs`/`maxEnemies` 같이 *없는 필드*를
만드는 환각 자주 발생.

| placeholder 필드 (정확) | 의미 | 사용처 |
|---|---|---|
| `SPAWN_CONFIG.firstSpawnDelayMs` | 게임 시작 후 첫 스폰까지 지연 (ms) | 게임 시작 시 `spawnTimer = -firstSpawnDelayMs`로 초기화 |
| `SPAWN_CONFIG.intervalByStage[stage]` | **현재 stage**의 스폰 주기 (ms) | `spawnTimer` 임계값 — stage 진행 시 자동으로 빨라짐 |
| `SPAWN_CONFIG.minOnScreen` | 화면 *항상* 유지할 최소 적 수 | 화면 적 < min이면 *주기 무관* 즉시 추가 spawn |
| `SPAWN_CONFIG.maxOnScreen` | 동시 등장 상한 | 도달 시 spawn skip |

🚫 환각 금지: `intervalMs`(단일값), `maxEnemies`/`maxCount`, `baseSpeed`(이건 ENEMY_MOVEMENT[key].speed). 모두 round retry 또는 stage 진행 무의미화.

### 9-2. 이동 패턴 — `ENEMY_MOVEMENT` 키 switch 필수 (3종 균등 확률)

**❌ 가장 흔한 버그** — LLM이 `.type` 같은 *존재하지 않는 필드*로 분기:
```js
if (movDef.type === 'zigzag') { … }   // .type 필드는 placeholder에 없음 → 항상 false
```
결과: 모든 적이 else 분기로 빠져 *직선 하강만* 됨.

**✅ 올바른 패턴** — placeholder 객체의 *키 자체*로 분기:
```js
// 1) 스폰 시: 키 무작위 선택, 문자열로 저장
const keys = Object.keys(ENEMY_MOVEMENT);          // ['straight','zigzag','formation']
const movementType = keys[Math.floor(Math.random() * keys.length)];
enemy.movementType = movementType;                  // 문자열

// 2) 이동 update: 문자열 키 자체로 switch
const m = ENEMY_MOVEMENT[enemy.movementType];      // { speed, … } 필드는 패턴별 다름
switch (enemy.movementType) {
  case 'straight':
    enemy.y += m.speed * dt;
    break;
  case 'zigzag':
    enemy._zigzagT = (enemy._zigzagT || 0) + dt;
    enemy.x += Math.sin(enemy._zigzagT * m.frequencyHz * 2 * Math.PI) * m.amplitude * dt;
    enemy.y += m.speed * dt;
    break;
  case 'formation':
    // 타원 편대 — 같은 _groupId·_groupIdx로 묶인 m.groupSize마리가 같은 타원 궤적
    // 코드 측이 ellipse 중심·반지름·각속도를 자유 설계 (placeholder는 groupSize만 명시)
    enemy.y += m.speed * dt;                       // y는 일관 하강
    // x는 ellipse phase (예: enemy._phase += angularSpeed * dt; x = cx + rx * cos(phase))
    break;
}
```

**필드명 정확히** (placeholder 환각 방지):
- `m.speed` (모든 패턴 공통)
- `m.amplitude`, `m.frequencyHz` (zigzag만) — `frequency` 같은 줄임 금지
- `m.groupSize` (formation만)

formation 구현 가이드:
- 스폰 시 *동시에 `m.groupSize`마리* 생성, 모두 같은 `_groupId` + 0..groupSize-1의 `_groupIdx` 부여
- 같은 ellipse 궤적 위에서 `_groupIdx`만큼 phase 어긋난 위치로 배치
- "여러 마리가 줄지어" 시각 효과 — 한 마리만 spawn하면 formation 의미 X

🚫 **금지 (모두 round retry 또는 시각 버그 유발)**:
- `movDef.type === '…'` 분기 (필드 없음)
- 'orbit'/'linear'/'wave' 등 *없는 키*
- formation 누락 (3종 중 1종이라도 빠지면 그 케이스 → 직선 fallback)
- `frequency` (placeholder는 `frequencyHz`)
- `enemySpeed`/`baseSpeed` 환각 (placeholder는 `ENEMY_MOVEMENT[key].speed`)

### 9-3. 공격 패턴 (행동·룰만, 구현은 LLM이 작성)

목적: *모든 적*이 자기 무기로 주인공을 향해 발사. *플레이어와 적이 같은 패턴 정의를
공유*해서 적별 분기·발산 방지. placeholder(`PATTERN_DEFINITIONS`)가 단일 진실 출처.

**행동 룰** (LLM이 이 룰대로 함수를 *직접 작성*):

1. **패턴 lookup** — 무기의 `pattern` 문자열(`'single'`/`'fan'`/`'pierce'`/`'omni'`)을
   key로 `PATTERN_DEFINITIONS`에서 `{description, bullets, pierce, speedMul, fixedDirection}` 조회.
   ⚠️ 적별 if/else로 패턴 분기 *절대 금지* — 모든 발사가 이 lookup 한 곳을 통과.
2. **placeholder 필드 구조 보존** — `bullets`는 *숫자 배열*(각도 offset, degrees).
   placeholder 그대로 `def.bullets.map(deg => …)` 로 순회. 객체 배열로 변환하거나
   wrap하지 말 것 (`rules/fe.md` 일반 룰 — placeholder field 타입 보존).
3. **기준 각도 결정** — 발사자별로 baseAngle을 다르게:
   - 플레이어: 위쪽 → `-Math.PI / 2`
   - 적: 발사 시점 player 위치를 향함 → `Math.atan2(player.y - enemy.y, player.x - enemy.x)`
   - `fixedDirection: true` (omni만): baseAngle 무시, bullets의 deg를 *절대 각도*로 사용
     → 구현 단순화: `const base = def.fixedDirection ? 0 : baseAngle`.
4. **각도 → 속도 벡터 변환** — 각 deg를 radian으로 변환해 `(vx, vy)` 계산:
   `rad = base + deg * π/180`, `vx = cos(rad) * speed`, `vy = sin(rad) * speed`.
   `speed = BULLET_SPEED * (def.speedMul || 1)`. `BULLET_SPEED` 도 placeholder.
5. **탄환 속성** — 각 탄환 객체에 다음 필드:
   - 위치: 발사자 위치(`sourcePos.x/y`)에서 시작
   - 속도: 위 #4의 `vx/vy`
   - 색: `weapon.color` (적은 `WEAPONS.find(w=>w.id===enemy.weaponId).color` — 무기 색과 동일)
   - 관통: `def.pierce` (true면 충돌 후에도 유지, 일반 탄환은 false → 충돌 시 제거)
6. **플레이어/적 공통 helper 권장** — 한 함수로 #1~#5 처리하고, 플레이어 발사와
   적 발사에서 *같은 함수 호출* (baseAngle만 다르게). 적별 별도 함수·분기 *금지*.
7. **발사 주기** — 적의 쿨다운은 `ENEMY_FIRE_COOLDOWN_MS[weaponId]` (ms) placeholder.
   플레이어의 쿨다운은 `WEAPONS[i].fireDelayMs` placeholder. *두 값은 다른 의미*
   (적은 자동 발사 주기, 플레이어는 연사 제한).
8. **스폰 직후 유예** — 적은 화면 진입 후 **0.5초**는 발사 안 함 (즉사 방지 UX).

🚫 **금지 패턴 (모두 round retry 유발)**:
- 적별 if/else 분기 (`if (enemy.weapon === 'fire_magic') { 부채꼴 } else if ...`)
- `bullets`를 객체 배열로 재구성 (`[{angleOffsetDeg: 0}]` → `deg * π/180`이 NaN)
- 각도 단위 혼동 (degree vs radian — placeholder는 degree, 변환 *후* radian 사용)
- 플레이어와 적이 *서로 다른 패턴 정의* (분기 발산)
